# Changelog

## Unreleased

- Generalized the `docker-logs` entry kind into `watched-command`, covering `docker`/`podman logs` (all their variants), `docker`/`podman compose up`, `kubectl logs -f`, `kubectl port-forward`, `tail -f`, `Get-Content -Wait`, `journalctl -f`, and `ssh`. **Breaking for existing saved state**: any `.vscode/terminal-state-manager.json` with an old `"kind": "docker-logs"` entry won't be recognized on the next restore (it just opens as an untracked terminal until a new command reclassifies it) — no state file migration was written since this hasn't shipped yet.
- Commands classified as a watched command now have their text redacted (`src/tracking/redact.ts`) before being written to `terminal-state-manager.json` or the debug log — inline secret-shaped env assignments, long-form secret CLI flags, URL-embedded passwords, and Bearer tokens are replaced with `***REDACTED***`.
- Tightened the `claude` and watched-command regexes to require a full word boundary (whitespace or end-of-string) rather than `\b`, which incorrectly treated a hyphen as a boundary too (`claude-monitor` was previously misclassified as a Claude session).
- Added a comprehensive vitest suite covering all of the above.
- Added Podman support (`podman logs`, `podman compose logs`, `podman-compose logs`, `podman container logs`, `podman compose up`/`podman-compose up`), sharing pattern-building logic with Docker.
- File-based debug logging (`.vscode/terminal-state-manager.log`) is now opt-in via `terminalManager.debugLogging` (default off) rather than always-on — it grows unboundedly with no rotation, so it isn't something to write into every user's workspace by default. The Output channel logging is unaffected (still always on, standard extension behavior).
- Performance: restoring many terminals at once is now staggered (`terminalManager.restoreStaggerMs`, default 150ms) instead of a tight creation loop; Claude session-file polling is now shared per distinct cwd (`src/claude/sessionFilePoller.ts`) instead of one independent poller per terminal; debug log writes are now async instead of blocking.
- Added 4 commands to toggle settings from the Command Palette instead of hand-editing `.vscode/settings.json`: **Enable**, **Disable**, **Enable Logging**, **Disable Logging**. Each writes to the workspace settings target specifically (never Global/User, to preserve per-project opt-in) and offers to reload the window.
- Added a one-time notification on activation while disabled, pointing at the **Enable** command directly instead of requiring the README to discover it exists. Suppression ("Don't show again") is workspace-scoped (`workspaceState`), not global.
- Added an extension icon (`images/icon.png`, referenced via `package.json`'s `icon` field) — a terminal prompt with a restore-arrow badge. Source SVG kept at `images/icon.svg` but excluded from the packaged `.vsix`.
- Documentation clarification (no code change): "workspace settings" means `.vscode/settings.json` for a plain folder open, but the `.code-workspace` file's own `"settings"` block if the project was opened via a saved/multi-root workspace file instead — `vscode.ConfigurationTarget.Workspace` resolves to whichever applies automatically. Confirmed via a real install where enabling via the notification wrote into a sibling `.code-workspace` file, leaving `.vscode/settings.json` untouched — correct behavior, but undocumented until now.
- Fixed Container Tools' (`ms-azuretools.vscode-containers`) "View Logs" action being tracked as `plain`. It runs as a VS Code Task, but (contrary to an initial assumption while investigating) a real `onDidStartTerminalShellExecution` event does fire for it -- the actual command is a PowerShell call-operator invocation of an absolute, quoted `docker.EXE` path (`& 'C:\...\docker.EXE' logs --tail 1000 -f <id>`), which the docker/podman regexes didn't recognize since they only matched a bare `docker`/`podman` at the start of the line. Broadened `enginePattern` in `classify.ts` to tolerate the call operator and a quoted/unquoted path prefix ending in the engine binary. Also added a name-based backup classifier (`classifyTerminalName`) recognizing a terminal named literally `Logs: <container>` and synthesizing a replayable `docker logs -f "<container>"`, for any similar case where no shell-execution event fires at all.
- Fixed `resumeMode` never reaching `"exact"` when actually resuming a session. Restoring an entry sends `claude --resume <session-id>` (an id we already know for certain), but that command getting typed reclassified the terminal as a *fresh* `claude` invocation, defaulting back to `interactive-fallback` and kicking off the same new-session-file poll used for a session that's never been seen before -- a poll that can never succeed here, since resuming appends to the *existing* session file rather than creating a new one, so it would always run the full ~5 minutes and time out. `extractResumeSessionId` in `classify.ts` now recognizes an explicit `--resume <uuid>` directly and skips polling entirely in that case.
- Diagnosed and mitigated a real "restore did nothing" report: closing and reopening the window landed on a `terminal-state-manager.json` with far fewer entries than were correctly written at close time, with a 13.5-minute gap in the debug log where our own code wasn't even running. Root cause: `.vscode/terminal-state-manager.json` was open in an editor tab; VS Code's hot-exit preserved a stale/dirty buffer across the window close and restored it on reopen, and that buffer being saved (Auto Save or manual) silently overwrote the extension's correct write, with no way for the extension to have prevented it since it wasn't running yet at that point. Added `StateRepository.fileUri()` plus a `vscode.workspace.onDidSaveTextDocument` listener in `terminalTracker.ts` that immediately reasserts the in-memory state if the file is ever saved externally while the extension *is* running -- doesn't close the gap for a save that happens before the extension activates, but does defend against the same thing happening mid-session. README now warns against leaving the file open with Auto Save on.
- Full pass to make every decision path automatically testable: extracted `parseState` (JSON-parse/shape-validate the loaded file) out of `storage.ts`'s I/O, `buildCorrelationResult` (the version-mismatch/not-found/found branching) out of `correlateClaudeSession`'s async orchestration, `needsClaudeCorrelation` out of an inline check in `terminalTracker.ts`, and `filterSessionFiles`/`parseClaudeVersionOutput` out of `listSessionFilesSync`/`getInstalledClaudeVersion`'s raw I/O. Added tests for all of these plus the previously-untested `sleep` and `createCancelToken`. 123 tests total; the only remaining untested code is now the minimal wrapper layer that calls real `vscode`/`fs`/`child_process` APIs directly, with no decision logic left in it to hide.
- Reload-adoption for same-named terminals now prefers matching by OS process id (`Terminal.processId`) over positional order, when a pid was recorded for a saved entry and a live terminal's pid matches it. VS Code persists the underlying process (and its pid) across a `Ctrl+R` reload, so this is a near-certain identity match where the old positional-queue approach could only guess by order. Doesn't help (and isn't attempted) for recreating terminals after a full close, since a freshly-spawned process always gets a new, unpredictable pid -- name is still the only signal there. `start()` is now async to allow resolving live pids before adopting.
- Added an integration test suite (`npm run test:integration`, `src/test/*.itest.ts`) running against a real headless VS Code instance via `@vscode/test-cli`/`@vscode/test-electron`, covering exactly the thin vscode-API glue the vitest unit suite structurally can't reach: real activation + command registration, a real terminal's open/close being tracked into the actual state file (including the window-close persist guard), real shell-execution events classifying a live PowerShell command, and `restoreStateNow` actually creating (and not duplicating) a real terminal from a pre-seeded state file. Uses a fixture workspace at `src/test/fixture-workspace` with tracking pre-enabled and auto-restore-on-startup disabled, so each test controls restore explicitly.
- Fixed Claude session correlation being permanently stuck at `resumeMode: "interactive-fallback"` for a terminal that was already in that state — a real, reproduced case (this very project's own terminal-state-manager.json). Root cause: correlation only ever waited for a *brand-new* `.jsonl` file to appear under `~/.claude/projects/<encoded-cwd>/`, but restoring an id-less `claude --resume` (the interactive-fallback path itself) resolves through Claude's own picker and just keeps *appending to an existing* session file — it never creates a new one. So a terminal already in fallback mode could never correlate, on any future restore: the "wait for a new file" poll would time out every single time, forever, no matter how many times it was reloaded. `listSessionFilesSync` now captures each file's mtime alongside its name, and `pickCorrelatedSessionFile` (renamed from `pickNewSessionFile`) now also matches on an *existing* file's mtime advancing past its "before" value, not just a new filename appearing — closing the loop for the resumed-session case the same way the new-file case already worked for a from-scratch `claude` invocation.
